최근 토스 채용 공고를 살펴보다가 토스에서는 현재 Deus라는 UI 에디터 툴을 만들고 있다는 사실을 알았다. 그 내용이 궁금해서 유튜브 영상도 찾아봤다. 보아하니 디자인 + 코딩 -> 제품까지 하나의 통합 툴에서 커뮤니케이션하고 유기적인 흐름으로 제품을 만들어 보자는 컨셉 같았다.

요즘은 Saas 제품이 너무 많고 심지어 툴마다 역할도 다양해서 어떤 걸 쓰느냐도 골치 아프다. 심지어 요금도 다 따로 부과돼서 만약 회사를 운영한다면 돈도 꽤 나갈 것 같다. 이런 측면에서 필요한 많은 기능이 합쳐져 있고 어느 정도 자유도만 부여된 제품이 있다면 써보고 싶다는 생각이 들 것 같다. Deus가 그랬다.

어쨌든, 예전부터 이런 툴의 필요성은 몸으로만 느끼고 머리로는 뭐라도 해 볼 생각조차 못 했다가 저 영상을 보고 자극을 받았다. 그래서 개인적으로 불편했던 경험을 토대도 나도 무언가를 한 번 만들어 보려고 한다.

불편함

불편함 1. 디자인 툴의 결과물과 내 코드와의 거리감.

나는 코드로 컴포넌트를 작성하고 스토리북이나 개발 서버를 돌려놓고 실시간으로 확인하면서 디자인하는 걸 좋아한다. 피그마 같은 디자인 툴은 디자인을 빠르게 뽑아낼 수 있어 좋지만 결국 이걸 다시 내 코드로 만들려면 시간이 걸리고 또 디자인 툴이 내 취향도 아니다.

불편함 2. 스토리북 환경

나는 스토리북 같이 내 코드로 디자인을 표현할 수 있는 게 더 편하게 느껴진다. 그런데 스토리북의 환경은 개발 환경과는 달라서 무언가를 추가하려면 설정을 계속해 줘야 한다. 예를 들어 tailwind를 적용하고 싶으면 새로운 모듈(@storybook/addon-styling-webpack)을 설치해야 하고 이런 수많은 의존성은 꾸준히 유지보수되기가 어렵다. 또 버그도 많다.

그래서 다음과 같이 생산성을 우선으로 하는 툴을 만들 것이다. (스토리북 기능 + 환경 통일) + (마우스로 컴포넌트 조작 + 실시간으로 내 코드에 반영) = ?

구현 아이디어 검증

  • 데스크탑 앱
  • 사용자 프로젝트 코드를 앱 데이터 폴더에 복붙한다.
  • 에디터 구현을 위한 맞춤 코드를 거기에 추가한다.
  • 자식 프로세스를 하나 만들어서 (npm || yarn || pnpm) dev를 돌린다.
  • iframe안에서 보여준다.
  • iframe 안팎으로 데이터를 주고받는다. 이 정도만 가능해도 구현은 가능할 것 같다.

데스크탑앱으로 만들 거라서 Tauri vs Electron 두 가지 선택지가 있었다. Electron은 Next.js라던지 Vite지원이 좀 애매하고 개발 세팅도 Tauri에 비해 복잡하다. 그렇지만 익숙한 Node.js환경이란 장점이 있다. 일단 예전에 조금 만져본 Tauri로 진행을 해봤다. ok

Tauri의 @tauri-apps/api모듈로 fs 작업은 자바스크립트로 가능해서 프로젝트 코드를 앱 데이터 폴더에 복사하는 건 가능하다. 그럼, 자바스크립트로 에디터 관련 코드를 추가하고 기존 코드 수정 또는 위치 변경도 가능하다. ok

Rust의 std::process::Command를 이용해서 복붙한 폴더에서 pnpm dev를 실행해 봤다. 잘 되는데 Tauri 프로세서가 죽어도 얘는 살아있다. unix 환경이라면 lsof -i :port; kill -9 pid;이런 명령어로 앱이 종료될 때 직접 죽여야 한다. tokio로 뭔가 해결해 볼 수 있을 것 같아서 좀 더 알아보려고 한다. ok

그런데 에디터 기능을 만들려면 메인 화면과 iframe간 통신이 필요한데 이게 좀 골치가 아팠다. iframe을 처음 만져보니까 어떻게 해야 할지도 잘 모르겠고 ChatGPT나 인터넷에 검색해 봐도 명확한 답이 없었다. window.postMessage로 전달하라는데 브라우저에서는 포트가 다르다고 오류만 나오고 결국 Tauri 문제인가 싶어서 Electron으로 넘어갔었다.

결국 똑같은 오류로 고생하다가 window.frames.postMessage("data", "*"), window.parent.postMessage("data", "*")로 하니까 어이없게도 잘 됐다. ok
